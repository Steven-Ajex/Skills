# 架构解读检查清单（Analysis Checklist）

在开始分析和输出结论前各检查一次，避免遗漏。

## A. 开始前检查

- 是否明确分析范围（全项目 / 子系统 / 链路）？
- 是否明确输出档位（快读 / 标准 / 深度）？
- 是否识别项目入口与启动方式？
- 是否扫描目录结构并形成初步模块分组？
- 是否找到配置、依赖装配、外部依赖接入点？
- 是否说明任何前提假设？

## B. 证据质量检查

- 每条重要结论是否有文件+行号证据？
- 是否区分事实 (Fact) 与推断 (Inference)？
- 关键推断是否标注置信度（高/中/低）？
- 是否避免仅凭命名推断行为？
- 是否至少追踪 1 条关键调用链？
- 若项目很大，是否声明抽样范围与未覆盖区域？

## C. 优点分析检查

- 是否说明“为什么是优点”，而不仅是现象描述？
- 是否指出优点的适用边界（在哪些条件下成立）？
- 是否覆盖至少 2 个维度（如边界、测试性、扩展性）？

## D. 风险分析检查

- 风险项是否带优先级（P1/P2/P3）？
- 是否给出触发条件与影响范围？
- 是否区分设计问题与代码风格偏好？
- 是否避免无证据的性能/并发判断？
- 是否识别高影响风险（边界、状态、错误处理、一致性）？

## E. 建议可执行性检查

- 是否按优先级分层（立即/中期/长期）？
- 每条建议是否包含目标与预期收益？
- 是否指出改动范围和潜在风险？
- 是否给出至少一种验证方式（测试/回归/日志对比）？
- 是否避免直接给出“大重构”但没有迁移路径？

## F. 输出风格检查（本技能约束）

- 是否使用中文为主？
- 专业名词首次出现是否附英文注释？
- 是否先给架构总览再给细节？
- 是否包含“不确定项与需确认问题”？
- 是否包含“残余风险/测试盲区”（深度版建议）？
