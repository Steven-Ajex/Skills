# 原子技能边界检查清单（Boundary Checklist）

在最终确定拆分方案前，逐项检查每个子技能。

## A. 目标边界

- 是否只有一个主目标（single primary goal）？
- 是否能用一句话清楚描述“负责什么”？
- 是否明确写出“不负责什么”？
- 是否定义了完成判据（Definition of Done）？

## B. 输入边界

- 输入数据类型是否清晰？
- 是否写明前置依赖（如果有）？
- 是否能在缺失前置依赖时明确报错或暂停？
- 是否能解释为什么这些输入是该技能不可再约简的最小输入？

## C. 输出边界

- 输出产物是否明确（表、结论、报告、代码、配置）？
- 是否有最小输出结构？
- 输出是否能被后续 skill 复用？
- 输出是否可被验证（不是纯口号式结论）？

## D. 触发边界

- 用户提问是否能明显触发该 skill？
- 是否会与其他 skill 高度重叠？
- 是否在 frontmatter description 中写清适用场景？
- 是否避免把端到端场景误写进原子技能 description？

## E. 专业边界

- 是否只使用一类主要专业方法？
- 是否跨了多个专业阶段（解码+分析+报告）？
- 如果跨阶段，是否应拆分或上移为编排技能？
- 是否体现该子技能的专业方法论，而不是通用流程套话？

## F. 维护边界

- 后续扩展该 skill 是否会显著扩大触发范围？
- 是否更适合作为新 skill 而不是继续膨胀现有 skill？

## G. 编排需求判断

- 该技能是否常与其他 2+ skills 固定顺序联用？
- 是否需要 workflow skill 来防止跳步？
- 是否需要在编排技能中汇总结论？

## H. 第一性原理检查（必须）

- 是否先定义了最小任务单元，而不是先想目录结构？
- 是否先定义输入/输出/完成判据，再定义技能名称？
- 是否能说明该技能存在的必要性（为什么不能并入其他 skill）？
