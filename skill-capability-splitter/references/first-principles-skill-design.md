# 第一性原理 Skill 设计框架（First-Principles Skill Design）

用于在创建新 skill 或优化旧 skill 时，避免从“目录、命名、模板”出发，而是先从任务本体出发。

## 1. 核心思想

一个高质量 skill 的本质不是“写了很多说明”，而是：

1. 对一个明确任务单元给出稳定的方法
2. 在给定输入下产出可验证输出
3. 降低误用和误触发

因此先定义任务本体，再设计技能壳层（名称、目录、模板）。

## 2. 四个不可跳过问题（4 Non-skippable Questions）

### Q1. 最小任务单元是什么？

问法：

- 用户真正要完成的最小工作是什么？
- 如果再拆，是否会失去独立可用性？

输出：

- 一句“最小任务单元定义”

### Q2. 输入、输出、完成判据是什么？

问法：

- 输入必须有什么？
- 输出必须是什么？
- 如何判断这个 skill 已完成且正确？

输出：

- `Input / Output / Definition of Done`

### Q3. 哪些前置依赖不可跳过？

问法：

- 如果跳过某步，会不会导致错误结论或低质量输出？

输出：

- “不可跳步列表”

### Q4. 方法论边界在哪里？

问法：

- 该任务需要的专业方法是什么？
- 是否与另一个任务的方法论本质不同？

输出：

- 专业方法边界说明（决定是否拆 skill）

## 3. 从第一性原理推导技能拆分

按顺序推导：

1. 定义最小任务单元
2. 列出多个任务单元（能力清单）
3. 检查输入/输出差异
4. 检查方法论差异
5. 检查是否存在固定前置依赖
6. 得出原子技能与编排技能结构

不要反过来做：

- 先按文件夹拆
- 先按章节拆
- 先按“感觉功能很多”拆

## 4. 高质量 / 高专业性 / 高可用性的具体含义

### 高质量（High Quality）

- 有完成判据
- 有边界声明
- 有可复用输出结构
- 有风险提示

### 高专业性（High Professionalism）

- 使用该任务专属方法论（不是泛化模板）
- 明确常见误判和排除路径
- 结论有证据或可验证条件

### 高可用性（High Usability）

- 用户容易触发正确 skill
- 输出结构稳定
- 前置依赖写清楚
- 与相邻技能关系明确

## 5. 设计时的常见偏差（Biases）

1. 模板偏差：先套模板再定义任务
2. 命名偏差：先想名字再想边界
3. 全能偏差：希望一个 skill 覆盖完整生命周期
4. 文档堆砌偏差：增加大量说明但没有提高完成率

## 6. 最终检查（Before You Freeze the Design）

在锁定 skill 设计前，确认：

1. 该 skill 的最小任务单元定义是否清晰
2. 输入/输出/完成判据是否完整
3. 是否有明确“不负责项”
4. 是否有专业方法说明
5. 是否与邻近 skills 的边界不重叠（或已声明重叠处理）
