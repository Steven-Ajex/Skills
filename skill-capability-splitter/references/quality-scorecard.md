# 拆分方案质量评分卡（Quality Scorecard）

用于评价“技能拆分方案”本身，而不是评价某个业务系统。

评分范围：1（差）到 5（优秀）

## 1. 触发准确性（Activation Precision）

看什么：

- 原子技能的 description 是否窄且准确
- 子问题是否不会误触发整套流程

低分表现：

- 多个原子技能描述高度相似
- 端到端场景写进原子技能 description

## 2. 边界清晰度（Boundary Clarity）

看什么：

- 每个 skill 是否写清“负责 / 不负责”
- 输入输出边界是否明确

低分表现：

- 原子技能之间都能做同一件事
- 编排技能重新实现原子技能核心方法

## 3. 专业性（Professional Depth）

看什么：

- 每个子技能是否体现专属方法论
- 是否包含该子任务常见误判与防护

低分表现：

- 全是通用流程词，没有专业判断框架

## 4. 可用性（Usability）

看什么：

- 用户是否容易选对 skill
- 输出结构是否稳定
- 前置依赖是否清楚

低分表现：

- 需要大量解释才能知道该用哪个 skill
- 输出结构高度随意

## 5. 可维护性（Maintainability）

看什么：

- 后续新增需求是否有明确落点
- 是否定义新增 vs 扩展规则

低分表现：

- 每次新增都不知道往哪里放
- 技能持续膨胀

## 6. 可验证性（Verifiability）

看什么：

- 每个子技能是否有完成判据（Definition of Done）
- 结果是否可检查而非主观表述

低分表现：

- 没有完成标准
- 只输出“建议进一步分析”类空话

## 7. 总评建议

### 评分解释（总分 / 30）

- `26-30`：可投入使用（高质量）
- `20-25`：可用但建议优化边界或触发描述
- `14-19`：存在明显问题，需重构部分拆分
- `<=13`：建议回到第一性原理重新定义任务单元

## 8. 改进动作模板

对低分项给出：

- 低分原因
- 影响
- 修正动作
- 是否需要新增 skill / 合并 skill / 重写 description
